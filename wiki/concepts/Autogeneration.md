# Autogeneration

## Definition

**Autogeneration** is the metadata-declared strategy by which a primary-key column gets its value when the caller of `create()` doesn't supply one. It is set per `@PrimaryColumn` via the `autogeneration` option, with two strategies (`clientSide` or `dbSide`).

```ts
type Autogeneration<Value> =
  | { clientSide: () => Value }
  | { dbSide: (sql: SQL) => SQL.Query<unknown> | undefined };
```

A `@PrimaryColumn` **without** `autogeneration` is a required field at compile time and at runtime — the caller must supply it.

## How It Works

### `clientSide`

The library invokes the supplied function **once before the INSERT**:

1. Call `autogeneration.clientSide()` to get a value (e.g., `crypto.randomUUID()`).
2. Write that value into the SQL statement's parameter list.
3. Include the same value in the `Partial<T>` returned from `create()`.

The database sees a normal `INSERT` with a fully-supplied PK — nothing special server-side.

### `dbSide`

The library **omits the column** from the `INSERT` entirely:

1. Build the SQL statement with the column missing from the column list.
2. Add the column to the `RETURNING` clause.
3. The database fills it (e.g., from a SERIAL sequence, a `gen_random_uuid()` default, or any expression the `dbSide` callback supplies).
4. Read the value back from the `RETURNING` row and put it in the `Partial<T>` result.

The `dbSide` callback's `(sql: SQL) => SQL.Query<unknown> | undefined` signature lets it produce a custom `DEFAULT` expression, but typical usage just returns `undefined` (rely on the column's existing `DEFAULT`).

> [!note] Why `SERIAL` for `dbSide` PKs
> `SERIAL` is the canonical `dbSide` type because PostgreSQL's `SERIAL` DDL emits an `INTEGER` column plus a sequence plus a `DEFAULT nextval(...)`. Foreign keys that reference this PK get the column type **automatically demoted to `INTEGER`** by `toForeignKeyType` — they don't claim their own sequence. See [[modules/sql-types]] § Foreign-key type demotion.

### Caller value always wins

If the caller passes a value for an autogenerated column, **the explicit value wins** (commit `cfe4cc5`). Passing `id: 42` to a SERIAL column is a supported override; `id: 'fixed-uuid'` to a UUID column is too. The autogeneration only fires when the field is absent.

## Why It Matters

### The explicit-only rule (commit `3aa354b`)

OOR previously inferred autogeneration from column type — SERIAL columns auto-generated; everything else didn't. Commit `3aa354b` removed that. Two reasons:

1. **Type and strategy are independent dimensions.** A `text` UUID column might be client-generated; an `integer` PK might or might not have a sequence. Conflating type with strategy was a leak.
2. **The compile-time `create()` story depends on metadata explicitly saying "omittable."** A SERIAL with no `autogeneration` is now a perfectly normal required field at the type level, and the runtime gate agrees — `create({ name: 'x' })` on such an entity is *both* a compile error and an `IncompletePrimaryKeyError` at runtime if the type check is bypassed.

This is what makes the type-level contract honest: there's no implicit "the database will sort it out" mode.

### The autogen-PK shape that made `update` silently broken (closed 2026-04-30)

Autogen PKs are declared with the `?` modifier — `id?: PrimaryKey<number>` — because `create()` allows the caller to omit them. That optional modifier on `T` is what made `update(entity: T)` accept `update({ name: 'x' })` and silently build `WHERE id = NULL`. The [[PrimaryKey Brand]] work closes this at the type level: `update`'s input is now `UnbrandedT<T> & PKInput<T>`, where `PKInput<T>` requires every PK key non-undefined regardless of the optional modifier on `T`. See [[sources/pk-aware-repository-methods]] and [[0008-pk-aware-compile-time]] for the full reasoning.

### Composite primary keys

For composite PKs, autogeneration is decided **per column**. One column can have `autogeneration: { dbSide: ... }`; another can be required. The `requirePrimaryKey` gate evaluates each independently.

## Examples

```ts
class User {
  // dbSide: SERIAL/sequence; column omitted from INSERT, returned via RETURNING.
  // Optional modifier (`?`) is required because `create()` lets the caller omit it;
  // PrimaryKey<number> brand is required by @PrimaryColumn's overload.
  @PrimaryColumn({
    type: COLUMN_TYPE.INTEGER,
    autogeneration: { dbSide: () => undefined },
  })
  id?: PrimaryKey<number>;

  @Column({ type: COLUMN_TYPE.TEXT }) @NotNullable
  email!: string;
}

class Session {
  // clientSide: library calls crypto.randomUUID() before INSERT.
  @PrimaryColumn({
    type: COLUMN_TYPE.TEXT,
    autogeneration: { clientSide: () => crypto.randomUUID() },
  })
  id?: PrimaryKey<string>;

  @Column({ type: COLUMN_TYPE.TEXT }) @NotNullable
  userId!: string;
}

class Account {
  // No autogeneration: caller MUST supply externalId.
  // Required modifier (`!`) and the brand are both required by the @PrimaryColumn overload.
  @PrimaryColumn({ type: COLUMN_TYPE.TEXT })
  externalId!: PrimaryKey<string>;

  @Column({ type: COLUMN_TYPE.TEXT }) @NotNullable
  name!: string;
}

const userRepo = new Repository(User, db);
const sessionRepo = new Repository(Session, db);
const accountRepo = new Repository(Account, db);

await userRepo.create({ email: 'a@b.com' });           // ok — id from DB
await sessionRepo.create({ userId: 'u1' });            // ok — id from clientSide
await accountRepo.create({ name: 'x' });               // compile error AND
                                                        // IncompletePrimaryKeyError —
                                                        // externalId is required.

// Caller value wins
await userRepo.create({ id: 42, email: 'a@b.com' });   // INSERT writes id=42
                                                        // explicitly; dbSide is skipped.
```

## Pitfalls

- **Forgetting `autogeneration` on a SERIAL.** If you migrate from another ORM where SERIAL implied auto-generation, your `@PrimaryColumn({ type: COLUMN_TYPE.INTEGER })` will be required at `create()` time. The error is loud (compile + runtime), but the cause may not be obvious.
- **`dbSide: () => undefined` as a no-op.** Many autogeneration cases just want "let the column's DDL `DEFAULT` handle it." The signature requires a function; passing `() => undefined` is the idiom for "I just want the column omitted from `INSERT`."
