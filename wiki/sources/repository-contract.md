# Repository Contract

## Summary

The pin-down note for `Repository<T>`: every operation, what each one owns, what the type system enforces at compile time, what the `requirePrimaryKey` gate enforces at runtime, the autogeneration strategies, the single error type, and what is deliberately out of scope. Closes the engineering picture across the five-source set: this source nails the *write side* in the same way `query-builder-design.md` nailed the read side.

## Key Claims

- **Operations:** `create(entity)`, `findById(key)`, `findOne(options)`, `findMany(options)`, `find()` (returns a `QueryBuilder<T>`, no SQL), `update(entity)`, `delete(key)`.
- **Single PK gate:** `requirePrimaryKey` is the private method every PK-needing operation funnels through. It decides "complete" based on `autogeneration` metadata; this is why the runtime story is symmetric across `create`, `findById`, `update`, and `delete`.
- **`create(entity: T)`** тАФ signature tightened from `Partial<T>` to `T` in commit `92ce5fd`. The argument's *type* already encodes the contract. The decorators shape `T` so:
  - `@NotNullable` field тЖТ required on `T` тЖТ required at `create()`.
  - `@Nullable` field тЖТ optional on `T` тЖТ optional at `create()`.
  - `@PrimaryColumn({ autogeneration })` тЖТ `NullableField<Value>` (optional).
  - `@PrimaryColumn` without `autogeneration` тЖТ `NotNullableField<Value>` (required).
- **Type-level rule (canonical phrasing):** *required on the type means required at `create()`*. The only escape is `autogeneration`.
- **`create()` return shape:** `Partial<T>` containing **only the primary-key fields**, populated from `RETURNING`. Not a hydrated entity. *"`create()`'s job is to commit a row and tell you how to find it again, not to re-read it."* Follow with `findById()` for the full row.
- **Autogeneration strategies** (see [[Autogeneration]]):
  - `clientSide: () => Value` тАФ library calls the function once before INSERT, writes into both the SQL and the returned key.
  - `dbSide: (sql) => SQL.Query<...>` тАФ column omitted from INSERT entirely; DB fills it; value returns via `RETURNING`.
  - An **explicit caller value always wins** over either strategy (commit `cfe4cc5`).
- **Explicit-only rule** (commit `3aa354b`): SERIAL columns no longer auto-generate implicitly just because of their type. Autogeneration is now a metadata flag, not a type inference.
- **Composite primary keys** follow the rule **per column** тАФ each PK column independently decides whether it's omittable.
- **Single repository error: `IncompletePrimaryKeyError`.** Raised by `findById` / `delete` (when key is partial), `create` (when a non-autogenerated PK is absent), and `update` (indirectly via `requirePrimaryKey`). Carries `entityName` and `missingProperties: string[]`.
- **Driver errors propagate unchanged.** The repository is intentionally thin around `pg` / Bun SQL тАФ wrapping their errors would obscure the diagnostic.
- **`find()` is a handoff** тАФ returns a builder, runs no SQL. `findOne` / `findMany` are sugar that build, apply options, and call the terminal in one shot.
- **Deliberately out of scope:**
  - **No partial updates.** `update(entity: T)` takes full `T`. Field-level merges belong in a builder, not a repository.
  - **No identity map.** Two `findById({ id: 1 })` calls may return two distinct objects. Equality is by value, not by reference.
  - **No cascade.** Relations are written when present in the input, ignored otherwise. The repository never reaches into another table on its own.
  - **No transaction management.** Caller's job (or a future concern). Repository takes a `Database` and uses whatever connection it hands.

## Corrections / Refinements to Existing Pages

- [[Repository Pattern]] тАФ the read/write split is sharpened to **by-key (single-row) vs. composed**. `create()` signature corrected to `T` (not `Partial<T>`); return type corrected to `Partial<T>` (PK fields only, not a hydrated entity). The "type shapes the contract" mechanism explained.
- [[ECMAScript Stage-3 Decorators]] тАФ the `@NotNullable` / `@Nullable` decorators don't just write `NULLABLE_KEY`; via the `NullableField<Value>` / `NotNullableField<Value>` types, they shape what `T` looks like at the type level. This is the link from runtime metadata to compile-time contract.

## Notes

The closing line of the source is the design statement worth carrying forward: *"These are not gaps; they are the boundary. Anything composed, reactive, or stateful goes elsewhere."* This sits alongside the query-builder source's *"keep the builder small, keep the SQL parameterized, keep the types honest"* as the pair of throughlines that should evaluate any future API addition to either layer.
