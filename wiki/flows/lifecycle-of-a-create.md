# Lifecycle of a `create()`

End-to-end walkthrough of a single insert. The orthogonal flows are [[query-lifecycle]] (composed reads) and [[schema-create-drop]] (schema-side); this is the per-row write path.

The example call:

```ts
await userRepo.create({ email: 'a@b.com', name: 'Alice' });
// returns: { id: 42 }   ← Partial<User> with only the PK
```

## Step 1 — Type check (compile-time)

The TypeScript compiler verifies the argument matches `T`. Per the entity's decorators, `T` already encodes the per-field optionality:

- `@NotNullable` columns are required.
- `@Nullable` columns are optional.
- `@PrimaryColumn({ autogeneration })` columns are optional (`NullableField<Value>`).
- `@PrimaryColumn` without autogeneration columns are required (`NotNullableField<Value>`).

Anything missing or mistyped is a compile error. **No runtime cost** — this step happens before the program is built.

See [[Repository]] § "create() — the contract in detail" for the table.

## Step 2 — Metadata lookup

The repository fetches the entity's resolved `EntityMetadata` from `db.getMetadata().get(EntityClass)` (see [[MetadataStorage]]). This is normally already-resolved by this point — first-read resolution has run during whichever earlier `Repository`/`QueryBuilder` call hit storage first.

From the metadata, the repository extracts:

- The full column list (for the `INSERT` column list).
- Each column's `nullable`, `primary`, and `autogeneration` flags.

## Step 3 — `requirePrimaryKey` gate (runtime)

For each `primary: true` column:

- If the column has `autogeneration` declared → fine, gate passes.
- If the column has no `autogeneration` and the entity object lacks the field → **throw `IncompletePrimaryKeyError`** with `entityName` and `missingProperties`.
- If the column has no `autogeneration` and the entity object provides the field → fine.

This is the same gate `findById`, `update`, and `delete` use — that's why the error story is uniform across the four methods.

## Step 4 — Resolve autogenerated values

For each PK column with `autogeneration`:

- **`clientSide`:** call the function (e.g., `crypto.randomUUID()`); store the value to write into both the SQL parameters and the returned PK.
- **`dbSide`:** mark the column as "omitted from INSERT" — it will not appear in the column list, but **will** appear in the `RETURNING` clause so the database-generated value comes back.

> [!note] Caller value wins (commit `cfe4cc5`)
> If the caller's `entity` object has a value for an autogenerated column, that explicit value wins. The autogeneration callback is **not** invoked, and the column appears in the `INSERT` like any normal one.

## Step 5 — Compose the `INSERT` SQL

Build a parameterized `INSERT ... RETURNING` statement using [[sqlJoin]] for the column list and the values list:

```sql
INSERT INTO "users" (email, name)
VALUES ($1, $2)
RETURNING id
```

Properties of this composition:

- **No string interpolation of user data** anywhere — values are bound parameters; identifiers go through `sql(c.columnName)`.
- **Columns with `dbSide` autogeneration are absent from the column list** but present in `RETURNING`.
- **`RETURNING` always names the primary-key columns**, so the result row carries the PK values regardless of which strategy filled them.

See [[Parameterized SQL]] and [[0004-parameterized-sql-only]].

## Step 6 — Execute

The composed `sql\`INSERT ... RETURNING ...\`` is awaited against [[Bun]]'s `SQL` driver. The result is a single row containing the PK values.

## Step 7 — Return

The repository assembles the returned `Partial<T>`:

- `clientSide` PK columns: the value computed in Step 4.
- `dbSide` PK columns: the value read from the `RETURNING` row.
- Caller-supplied PK overrides: the caller's value (already in the `INSERT`, also returned via `RETURNING`).

The returned object has **only PK fields**. The non-PK columns the caller passed are not echoed back. This is deliberate: *"`create()`'s job is to commit a row and tell you how to find it again, not to re-read it."*

If you want the full row, the next call is `repo.findById(returnedKey)`.

## Failure Modes

| Step | What | Outcome |
|---|---|---|
| 1 | Required field missing or mistyped | Compile error — never runs |
| 3 | Non-autogenerated PK absent (untyped caller) | `IncompletePrimaryKeyError` with `missingProperties` |
| 4 | `clientSide` callback throws | The exception propagates unchanged — repository doesn't catch |
| 6 | Constraint violation, FK error, connection drop | `pg` / Bun SQL error propagates unwrapped |
| 6 | UNIQUE violation on a caller-supplied PK | Same — driver error reaches the caller |

The repository is intentionally thin around the driver's own errors; wrapping them would obscure the diagnostic.
